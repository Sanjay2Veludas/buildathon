# Feasibility + Scope Plan — Audio-Localizing AI Monitor

## 1) Executive feasibility verdict
**Verdict: Feasible for a hackathon MVP in 5 hours with 3 devs** if scope is staged and demo-first.

- **Core loop feasibility (high):** sound trigger → coarse bearing → turret point → capture image/audio → local/cloud classify → speak + LED/LCD action.
- **Primary constraints:** App Lab RPC integration details, USB OTG stability (camera+speaker), and real-time noise handling.
- **Success condition:** reliable cloud-connected demo path with stable network and predictable response timing.

---

## 2) Architecture feasibility assessment

### A. MCU side (STM32U585 sketch)
**Feasible:** ADC sampling + event extraction + stepper control + UI IO are straightforward on U585.

- 3 analog channels at 1kHz each (3kS/s total) is lightweight.
- Rolling baseline + peak window (100ms) is low compute.
- Stepper with accel/decel + angle limit ±180° is feasible with software angle tracking.
- Trigger mute while motor moves + 300ms post-motion is essential and feasible.

**Risk:** exact RPC API shape for Arduino App Lab/Router; mitigate by isolating bridge handlers and having serial fallback command parser.

### B. Linux side (QRB2210 Debian Python)
**Feasible:** webcam capture, arecord capture, local state machine, timeout-wrapped cloud calls are standard.

- OpenCV frame grabs from `/dev/video*` are routine.
- ALSA capture/playback with C270 mic + USB speaker is routine if devices pinned by name.
- Anthropic call with strict JSON prompt and robust parsing is feasible.
- Cloud-first flow is feasible with strict timeout control and clear operator-visible error handling.

**Risk:** USB enumeration instability; mitigate with subsystem smoke tests and cached device IDs.

### C. End-to-end timing
**Feasible demo timing target:**
- Trigger detect: <150ms
- Turret point: 0.7–2.0s (depends on motor/steps)
- Capture: photos immediate, audio 3s
- Cloud analysis: timeout at 8–10s
- Total response: ~5–12s in online mode (network + cloud dependent)

## 3) Recommended product modes (operational modes)

## 1) Standby Mode
- Trigger processing off, turret idle, low-power visual status.
- Enter via button or RPC `set_armed(false)`.
- Use for setup/safety.

## 2) Armed Monitoring Mode (default demo mode)
- Continuous 3-channel sensing + adaptive baseline.
- Trigger threshold runtime-tunable.
- On trigger: emit event packet with bearing + amplitudes.

## 3) Event Localization Mode
- Latched event state for short window.
- MCU computes coarse bearing from two loudest sensors.
- Linux starts audio capture immediately and commands turret point.

## 4) Capture Mode
- Collect 2 still images + 3s audio clip.
- Works even if cloud disabled.
- Save evidence bundle locally with timestamp.

## 5) Analysis Mode (Online-only)
- Cloud multimodal inference is mandatory for verdict generation.
- Strict JSON parse with defensive recovery and retry policy.
- On timeout/error, emit explicit "analysis unavailable" state and return to monitoring.

## 6) Response Mode
- TTS output + LED/LCD action tier:
  - benign: blue status + short quip
  - notable: store artifacts + spoken summary
  - alert: red blink + alarm phrase

## 7) Demo Recovery Mode
- Handles missing camera/mic/network with fast operator-visible diagnostics.
- No offline classification path; system reports degraded state and waits for recovery.
- Critical for stage reliability in online-only scope.

## 8) Service/Calibration Mode
- Live threshold tuning, baseline readouts, manual turret jog.
- Sensor health checks and event counter reset.

## 9) (Optional) Investigate Motion Mode (stretch)
- Drive chassis toward bearing using DC motors + ultrasonic stop.
- Must be disabled by default behind feature flag.

## 5) Risk register with mitigations (demo reliability focus)

1. **OTG hub/camera not enumerating**
   - Mitigation: `test_camera` first; fallback to previous cached frame + audio-only response.
2. **Venue noise causing false triggers**
   - Mitigation: runtime threshold + hysteresis + trigger refractory period.
3. **Motor self-triggering sensors**
   - Mitigation: sensor mute during motion + 300ms cooldown (mandatory).
4. **Wi‑Fi/cloud outage**
   - Mitigation: strict timeout, one bounded retry, operator-visible fault state, and rapid re-arm when connectivity returns.
5. **Brownout/reboot when motor starts**
   - Mitigation: acceleration ramps, serialized motor actions, no simultaneous high-current starts.
6. **RPC mismatch under time pressure**
   - Mitigation: thin bridge abstraction + mock bridge for local testing.

## 8) Coding scope decisions to lock now

1. **Default runtime mode:** Armed Monitoring with online analysis required.
2. **Cloud is dependency:** verdict/action classification only from API response.
3. **Single event in-flight:** ignore/requeue new triggers during CAPTURING/ANALYZING.
4. **Static home calibration:** boot angle = 0°, no active homing hardware in MVP.
5. **Turret limit hard clamp:** enforce ±180° in MCU regardless of Linux command.
6. **Config-driven behavior:** all thresholds/timeouts/model/feature flags in `config.yaml`.

---

## 9) Acceptance criteria for hackathon demo

- Device detects loud clap and points turret roughly toward source.
- Captures media and provides spoken verdict within 12s in normal network conditions.
- LED/LCD state transitions visibly match state machine.
- If camera or internet fails, device enters explicit fault state on LCD/LED, announces analysis unavailable, and re-arms automatically after backoff.
- No brownout resets during repeated trigger cycles.

## B) Recommended pin map (default for code scaffold)

### Analog sound sensors
- Sensor S0 (0° bearing)  -> `A0`
- Sensor S1 (120° bearing)-> `A1`
- Sensor S2 (240° bearing)-> `A2`
- Sensor VCC -> 3.3V rail
- Sensor GND -> common GND

### Stepper driver (STEP/DIR profile, preferred)
- `D2` -> `STEP`
- `D3` -> `DIR`
- `D4` -> `EN` (optional, active-low typical)
- Driver `VMOT` -> battery + (7.4V)
- Driver `GND` -> common GND
- Driver logic `VIO` -> 3.3V (if required)

### Alternate stepper driver (ULN2003 + 28BYJ-48 profile)
- `D2` -> `IN1`
- `D3` -> `IN2`
- `D4` -> `IN3`
- `D5` -> `IN4`
- ULN2003 `VCC` -> 5V (or driver-required supply)
- ULN2003 `GND` -> common GND

### LEDs (2 red, 2 blue)
- `D6` -> RED_1 (through 220Ω resistor)
- `D7` -> RED_2 (through 220Ω resistor)
- `D8` -> BLUE_1 (through 220Ω resistor)
- `D9` -> BLUE_2 (through 220Ω resistor)
- LED cathodes -> GND

### Buttons (arm/standby, demo-reset)
- `D10` -> BTN_ARM (INPUT_PULLUP, switch to GND)
- `D11` -> BTN_RESET (INPUT_PULLUP, switch to GND)

### LCD 16x2 (choose one via config)

#### Option 1: I2C backpack (recommended)
- `SDA` -> board SDA pin
- `SCL` -> board SCL pin
- `VCC` -> 5V
- `GND` -> GND

#### Option 2: Parallel 4-bit mode
- `RS` -> `D12`
- `E`  -> `D13`
- `D4` -> `D14`
- `D5` -> `D15`
- `D6` -> `D16`
- `D7` -> `D17`
- `VSS` -> GND, `VDD` -> 5V, `VO` -> contrast pot wiper
- `A/K` backlight -> 5V/GND (per module)

### Optional 7-segment (via TM1637 suggested)
- `CLK` -> `D18`
- `DIO` -> `D19`
- `VCC` -> 5V
- `GND` -> GND

## D) Power wiring (critical)
1. Battery `+` -> UNO Q `VIN` and motor driver `VMOT`.
2. Battery `-` -> system ground bus.
3. UNO Q GND, motor driver GND, sensor GND, LCD GND, button GND all common.
4. Do not route motor current through thin signal ground traces; star-ground near battery negative.
5. Add bulk capacitor near motor driver (e.g., 470–1000uF) + local decoupling.

## E) USB topology
```text
UNO Q USB-C <-> OTG Hub <-> Logitech C270
                        <-> USB Speaker
```
- Keep short quality cables.
- Verify both enumerate before demo run.

## F) Safety and logic-level constraints
- Use only maker headers (JDIGITAL/JANALOG).
- Do not use 1.8V processor headers.
- Keep MCU GPIO at 3.3V logic compatibility for driver inputs.
- Enforce turret software clamp at ±180°.

## G) Bring-up wiring checklist
1. Confirm common ground continuity.
2. Verify sensor idle ADC values are stable.
3. Verify button reads with pull-ups.
4. Jog stepper at low speed first, confirm direction.
5. Verify LEDs and LCD independently.
6. Only then connect camera + speaker via OTG hub and run smoke tests.
